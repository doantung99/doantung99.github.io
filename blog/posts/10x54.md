## Summary

`10X54` ships a 3 MB ESP32-S3 Arduino firmware image (`ducktricks.bin`) that, on a real board, prints `KEY>`, reads a 16-byte password over USB serial, and either prints `GRANTED` or `DENIED`. The flag is decrypted on the GRANTED path with `flag[i] = i ^ cipher[i] ^ key[i%16]`, where `key` is a 16-byte Linear-Congruential-Generator (LCG) keystream seeded from the correct input. The trick: you never have to solve the password checks — an LCG is fully invertible from a handful of known output bytes, so a four-byte `V1T{` crib pins the 32-bit state and decrypts the whole flag.

This one was a collaboration. I recognized the shape of the problem and pointed the model at the right artifacts; the LLM did the disassembly grind, the byte-order bookkeeping, and the brute force. My job was steering: feeding it the right segment, catching when it chased decoys, and forcing it to justify the LCG inversion before I trusted the output.

## Solution

### Reading the terrain

The first thing I did was *not* dump it on the model raw. A 3 MB blob with no framing is a great way to get a confidently wrong answer. I ran the recon myself and gave the model the structured facts.

`strings ducktricks.bin` is almost entirely ESP-IDF / Arduino boilerplate (project `arduino-lib-builder`, ESP-IDF v5.5.4, USB-CDC console). No `v1t{`, no `flag`, nothing. That absence is itself a signal: the app identifies itself only by behavior, and any human-readable UI strings must be obfuscated.

`esptool image_info` is the load-bearing tool here. Despite the 3 MB file size, only the first ~286 KB is a *valid* ESP32-S3 image: six segments plus a footer that checksums clean. The two appended regions at file offsets `0x46000` and `0x88000` are **decoy Xtensa code blobs** — they look like more firmware, they disassemble to plausible instructions, and they contain no flag. This is the first dead-end the challenge plants, and it's exactly the kind of thing an over-eager LLM will tunnel into ("here's more code, let me analyze it!"). I noted it and told the model to ignore everything past the validated footer.

The valid image's segments:

```
seg0 DROM  0x3c030020   rodata / strings   <- the interesting one
seg1 DRAM  0x3fc96f00
seg2 IRAM  0x40378000
seg3 IROM  0x42000020   the bulk of the code
seg4 IRAM  0x40383154
seg5 RTC   0x50000200
```

Two things matter for everything downstream: **DROM at `0x3c030020`** holds the strings/constants, and **IROM at `0x42000020`** holds the code. Knowing those two base virtual addresses lets you map any "flat" file offset the disassembler reports back to a load address, and vice versa. I kept a sticky note of those two numbers because every later step depends on translating between them.

### The `^0x42` string obfuscation

Right at the start of `.rodata` (vaddr `0x3c030120`) there's a 60-byte blob that is clearly not printable text. The question is what consumes it. I had the model trace cross-references into that address, and it landed on a tiny helper at flat `0x1a4c`:

```
0x1a4c: l8ui a11, a2        ; load one byte from the blob
        movi a8, 66         ; 0x42
        xor  a11, a11, a8   ; byte ^ 0x42
        ...                 ; Serial.write(a11)
```

That's the whole secret to the UI strings: every printed character is stored XORed with `0x42`. This is a *textbook* ESP32 firmware string-hiding pattern, which is why it's worth recognizing on sight rather than reverse-engineering from scratch. Batch-decoding the blob with `^0x42`:

```
0x120 -> "DENIED\r\n"
0x128 -> "GRANTED\r\n"
0x131 -> "KEY> "
0x136 -> "VaultPass v1.3\r\n"
```

So the app is **"VaultPass v1.3"**: a serial crackme that prompts `KEY>`, reads **16 bytes**, validates them through a state machine, and prints `GRANTED`/`DENIED`. Crucially, finding `GRANTED` and `DENIED` tells you exactly where to set your sights — the flag lives on whatever code path emits `GRANTED`.

### The validation state machine (and why I refused to solve it)

`setup()` is at flat `0x1a6c`. After reading the 16 input bytes it runs a layered check:

- an **XOR checksum** of the input bytes compared against `0x57`,
- a **CRC** compared against the constants `0xfdd0` / `0x9456`,
- and an **LCG** keystream derived from the input.

This is a rabbit hole. You *could* model all three constraints and solve for a valid 16-byte password — and that is precisely the intended-but-expensive path the challenge dangles in front of you. I made an early judgment call here and told the model in no uncertain terms: **do not try to satisfy the password.** We only care about the password insofar as it seeds the keystream, and we have a cheaper handle on the keystream than solving for the seed. Holding that line was the single most important decision in the solve. Every minute spent constraint-solving the CRC was a minute wasted.

### The flag-decrypt routine

On the GRANTED branch (flat `0x1c11`) the firmware assembles a *cipher buffer* with three `memcpy`s, then runs the decrypt loop at flat `0x1c48`:

```
flag[i] = i ^ cipher[i] ^ key[i % 16]      for i in 0..37   (38 bytes)
```

Two unknowns: the exact byte order of `cipher`, and the 16-byte `key`. Both are recoverable from static analysis alone.

**cipher — the blob's thirds, reversed.** The three `memcpy`s into `stack+54` don't copy the source blob in order. The source is the 38-byte region `d[0x146:0x16c]` (these are DROM offsets — i.e. relative to the `.rodata` dump). The copies reassemble it as:

```
cipher = d[0x160:0x16c]   # 12 bytes  -- last third, placed first
       + d[0x153:0x160]   # 13 bytes  -- middle third
       + d[0x146:0x153]   # 13 bytes  -- first third, placed last
```

So `cipher` is the blob with its three thirds **reversed in order** (each third's internal bytes stay forward). This is the second place a careless reader goes wrong: it's easy to assume `cipher == d[0x146:0x16c]` and get garbage out. I made the model state the offsets explicitly and confirm the three lengths summed to 38 before going further. They do (12 + 13 + 13).

**key — an LCG keystream.** The 16-byte `key` buffer is written in exactly one place, by a Linear-Congruential Generator:

```
x = (x * 0x6c62272e + 0x07354a6b) & 0xffffffff
key[i] = (x >> 24) & 0xff               for i = 0..15
```

The multiplier `0x6c62272e` and increment `0x07354a6b` are the FNV-1 prime and offset basis, reused here as LCG parameters — a nice tell that these are "borrowed" magic constants rather than something derived from the input. The *seed* `x` is derived from the correct input, so we don't get it directly. But we don't need it.

### Recovering the key without the password — the key insight

Here is the crux of the whole challenge. An LCG is a deterministic, fully reversible 32-bit state machine. Each `key[i]` is the **top byte** `(x >> 24)` of a successive state. If I know even a few consecutive output bytes, I know the high 8 bits of several consecutive states, and that's enough to pin the entire 32-bit state by brute-forcing only the unknown low bits.

Where do known output bytes come from? From a **crib on the flag**. The flag must start `V1T{`. Plug that into the decrypt equation backwards:

```
key[i] = i ^ cipher[i] ^ flag[i]      for the first 4 bytes
```

That yields `key[0..3]` — four consecutive top-of-state bytes from the LCG. Now brute-force the low 24 bits of the first state `x1`:

- guess `x1 = (key[0] << 24) | lo` for each `lo` in `0..2^24`,
- step the LCG forward and require `(x2 >> 24) == key[1]`, then `key[2]`, then `key[3]`,
- the survivor (there's essentially one) is the true first state.

From that state, regenerate all 16 keystream bytes forward and decrypt the full 38-byte flag. `2^24` ≈ 16.7M iterations with three early-exit checks runs in seconds in Python.

The gotcha that actually mattered: **the crib must be uppercase `V1T{`, not lowercase `v1t{`.** The flag-format banner says `v1t{...}`, but this organizer's hardware track prints uppercase `V1T{`. Only the uppercase crib produces a single LCG state whose *full* keystream decrypts all 38 bytes to clean ASCII. The lowercase crib yields either no surviving state or one that decodes to garbage past byte 4. I caught the model trying lowercase first; the tell was that bytes 5+ came out as non-printable noise. The verification rule "the whole thing must be printable ASCII" is what disambiguates — a four-byte crib match is necessary but the full-decrypt sanity check is what confirms you found the *right* state.

### End-to-end script

This takes the raw `ducktricks.bin` to the printed flag. It parses the ESP image to extract the `.rodata` (DROM) segment, slices the reversed-thirds cipher, applies the `V1T{` crib, cracks the LCG, and decrypts.

```python
#!/usr/bin/env python3
# Solve V1t CTF 2026 "10X54" — ESP32-S3 VaultPass crackme.
# Recovers the flag by cracking the LCG keystream from the V1T{ crib;
# the 16-byte password is never solved.

import struct

FIRMWARE = "ducktricks.bin"

# --- ESP32-S3 image parse: find the DROM (.rodata) segment ----------------
# Image header: magic(1)=0xE9, segment_count(1), flash_mode(1), flash_size/freq(1),
# entry(4). Then per segment: load_addr(4), length(4), data[length].
DROM_VADDR = 0x3c030020          # where seg0 (.rodata) loads

def extract_drom(path):
    with open(path, "rb") as f:
        blob = f.read()
    assert blob[0] == 0xE9, "not an ESP image"
    nseg = blob[1]
    off = 8                       # skip 8-byte image header
    for _ in range(nseg):
        load_addr, length = struct.unpack_from("<II", blob, off)
        off += 8
        data = blob[off:off + length]
        off += length
        if load_addr == DROM_VADDR:
            return data
    raise RuntimeError("DROM segment not found")

d = extract_drom(FIRMWARE)        # d[x] == byte at DROM offset x

# --- cipher = the 38-byte blob with its three thirds reversed -------------
cipher = d[0x160:0x16c] + d[0x153:0x160] + d[0x146:0x153]
assert len(cipher) == 38, len(cipher)

# --- LCG parameters (FNV-1 prime / offset basis, reused as LCG) -----------
M, C, MASK = 0x6c62272e, 0x07354a6b, 0xffffffff

# --- crib: flag starts "V1T{" (UPPERCASE) -> recover key[0..3] ------------
crib = "V1T{"
k = [i ^ cipher[i] ^ ord(ch) for i, ch in enumerate(crib)]

# --- brute the low 24 bits of the first LCG state, gated by the crib ------
def crack():
    for lo in range(1 << 24):
        x1 = (k[0] << 24) | lo
        x2 = (x1 * M + C) & MASK
        if (x2 >> 24) != k[1]:
            continue
        x3 = (x2 * M + C) & MASK
        if (x3 >> 24) != k[2]:
            continue
        x4 = (x3 * M + C) & MASK
        if (x4 >> 24) != k[3]:
            continue
        # candidate state: regenerate full 16-byte keystream and decrypt
        ks, x = [(x1 >> 24) & 0xff], x1
        for _ in range(15):
            x = (x * M + C) & MASK
            ks.append((x >> 24) & 0xff)
        flag = bytes(i ^ cipher[i] ^ ks[i % 16] for i in range(38))
        if all(32 <= b < 127 for b in flag):   # full-decrypt sanity gate
            return flag, bytes(ks)
    return None, None

flag, keystream = crack()
print("key      =", keystream.hex())
print("flag     =", flag.decode())
```

Running it recovers the 16-byte keystream `7ab01cdbe475f2174a298fd5da73b7ad` and prints the flag. The full-decrypt-to-ASCII gate is what makes the result trustworthy rather than just a 4-byte coincidence.

## Lessons learned - prompting the AI

The model can disassemble Xtensa and grind a `2^24` brute force far faster than I can. What it cannot do reliably is *choose the cheap path* and *not get seduced by decoys*. My value here was direction and verification, and the prompting reflects that.

**1. Frame the goal as "recover the flag," not "solve the challenge."** The single best move was forbidding the password solve up front:

> "This is an ESP32-S3 crackme. The flag is decrypted as `flag[i] = i ^ cipher[i] ^ key[i%16]` where `key` is a 16-byte LCG keystream `x = (x*0x6c62272e + 0x07354a6b) & 0xffffffff; key[i] = x>>24`. DO NOT try to satisfy the XOR-checksum / CRC password checks. Instead recover the LCG state directly from a known flag prefix, then regenerate the keystream and decrypt."

This kills the most expensive dead-end (constraint-solving the CRC) before the model spends a single token on it.

**2. Make it commit to the byte layout in writing before computing.** The reversed-thirds cipher is the easiest place to silently get garbage:

> "List the exact DROM offsets and lengths of every memcpy into the cipher buffer, in copy order. Confirm the lengths sum to 38 and reconstruct the slice expression. Do not assume the source is copied in order."

Forcing it to *state offsets and check the length sum* turns a subtle bug into a visible, checkable claim.

**3. Pin the crib case and demand a full-decrypt verification.** The model defaulted to lowercase `v1t{` (because the flag format says so) and the brute force returned a state that decoded to noise after byte 4. The fix was a verification rule, not a guess:

> "Try the crib as uppercase `V1T{`. Reject any LCG state unless ALL 38 decrypted bytes are printable ASCII — a 4-byte crib match alone is not sufficient proof."

That rule is how I caught the wrong turn: I didn't trust the 4-byte match; I trusted only the fully-printable 38-byte output. When lowercase produced garbage past byte 4 and uppercase produced clean text ending in `}`, the ambiguity resolved itself.

**What to tell the model to focus on:** the `GRANTED` code path (that's where decryption lives), the single function that writes the `key` buffer (that's your LCG), and the decrypt loop's exact index math (`i ^ cipher[i] ^ key[i%16]`). **What to tell it to avoid:** the appended decoy blobs past the validated image footer (file offsets `0x46000` / `0x88000`), and any attempt to forge a valid password.

**How I verified:** I cross-checked the two segment base addresses (`0x3c030020` DROM, `0x42000020` IROM) myself with `esptool image_info` so the model's offset translations had ground truth; I re-derived the `^0x42` decode by hand on the `KEY> ` string to confirm the helper; and I gated the brute-force result on full printable-ASCII output rather than the crib match alone.

**Fast-path prompt recipe for next time:** *"Firmware crackme — give me the flag, not the password. Find the GRANTED decrypt loop, identify the keystream generator (LCG/LFSR/PRNG), invert its state from a known flag-prefix crib, and accept the answer only if the entire decrypted buffer is printable ASCII."*
